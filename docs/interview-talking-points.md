# Interview Talking Points

## Short Pitch

My game-systems work is independent rather than studio employment, but it is deep runtime work. I
have 5+ years of Android game modding experience, including Unity IL2CPP, Cocos2d-x/Lua, native C++
hooks, LSPosed/Xposed modules, engine detection, movement/collision/game-loop analysis, and emulator
automation.

## Why It Fits This Role

This role is about game engines and AI research. My strongest work is exactly at the boundary between
runtime environments and automation:

- instrumenting games where no clean API exists
- extracting or verifying state from real app runtimes
- accelerating repetitive simulation loops without dropping side effects
- building tools that make runtime behavior observable and testable
- setting up Android emulator/root workflows for repeatable testing

## How To Frame The Experience

Use "independent Android game runtime systems work" rather than "game studio experience."

Good wording:

> Alongside my AI model evaluation contract work, I have 5+ years of independent Android game modding
> and runtime systems experience. My work includes Unity IL2CPP analysis, LSPosed/Xposed Android
> modules, native C++ hooks, metadata-based method and field resolution, emulator/root tooling, and
> movement/collision/game-loop modification.

## If Asked About Godot / Defold / Stride

Be precise:

> I have not shipped a Godot or Defold game from the editor, but I have worked on engine-runtime
> instrumentation across Android game engines and built detection/routing infrastructure for Unity,
> Unreal, Cocos2d-x, Godot, and related native-heavy runtimes. My experience is deeper on the runtime
> and systems side than on editor-level asset production.
